# 技术决策记录

本文档记录 TaskManager 项目的重要技术决策和架构选择。

## ADR-001: 采用分层架构模式

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

需要设计一个可扩展、易维护的任务管理系统架构。

### 决策

采用经典的分层架构模式：
- 表示层 (CLI)
- 业务层 (Models)
- 存储层 (Storage)
- 工具层 (Stats)

### 后果

**正面影响：**
- 职责清晰，模块解耦
- 便于单元测试
- 支持多种存储后端
- 易于扩展新功能

**负面影响：**
- 代码层次较多
- 简单功能可能过度设计

**缓解措施：**
- 保持接口简洁
- 避免过度抽象

## ADR-002: 使用抽象基类定义存储接口

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

需要支持多种存储后端（内存、JSON文件、可能的数据库），同时保持业务逻辑与存储实现解耦。

### 决策

使用 Python ABC (Abstract Base Class) 定义 `BaseTaskStore` 接口：

```python
class BaseTaskStore(ABC):
    @abstractmethod
    def add(self, task: Task) -> str: ...
    @abstractmethod
    def get(self, task_id: str) -> Optional[Task]: ...
    # 其他抽象方法...
```

### 替代方案

1. **Protocol类型** - 更轻量，但缺少运行时检查
2. **鸭子类型** - 灵活但缺少接口约束
3. **直接继承** - 简单但不够灵活

### 后果

**正面影响：**
- 强制接口一致性
- 运行时类型检查
- IDE支持更好
- 便于Mock测试

**负面影响：**
- 增加代码复杂度
- 必须实现所有抽象方法

## ADR-003: 选择JSON作为默认持久化格式

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

需要选择一种数据持久化格式，要求简单、可读、跨平台兼容。

### 决策

使用JSON格式存储任务数据到本地文件。

### 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **JSON** | 可读性好、跨平台、标准库支持 | 性能一般、不支持复杂查询 |
| **SQLite** | 功能强大、查询灵活、ACID | 学习成本高、过于复杂 |
| **Pickle** | 性能好、Python原生 | 不可读、安全风险、不跨语言 |
| **YAML** | 可读性极好、支持注释 | 解析复杂、性能差 |
| **CSV** | 简单、Excel兼容 | 结构限制、不支持嵌套 |

### 后果

**正面影响：**
- 数据可读性好，便于调试
- 跨平台兼容
- 标准库支持，无额外依赖
- 容易备份和版本控制

**负面影响：**
- 大数据集性能不佳
- 不支持复杂查询
- 并发写入可能有问题

**缓解措施：**
- 数据量大时可切换到SQLite
- 实现文件锁防止并发问题

## ADR-004: 采用dataclass定义数据模型

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

需要定义任务相关的数据结构，要求类型安全、易于序列化。

### 决策

使用 Python 3.7+ 的 `@dataclass` 装饰器定义数据模型：

```python
@dataclass
class Task:
    title: str
    status: TaskStatus = TaskStatus.TODO
    priority: Priority = Priority.MEDIUM
    created_at: datetime = field(default_factory=datetime.now)
```

### 替代方案

1. **普通类** - 需要手写大量样板代码
2. **namedtuple** - 不可变，缺少方法支持
3. **dict** - 缺少类型约束和IDE支持
4. **Pydantic** - 功能强大但增加依赖

### 后果

**正面影响：**
- 自动生成 `__init__`、`__repr__` 等方法
- 类型注解支持
- IDE 自动完成
- 代码简洁

**负面影响：**
- Python 3.7+ 要求
- 一些高级特性需要手动实现

## ADR-005: 使用UUID作为任务ID生成策略

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

任务需要唯一标识符，要求全局唯一、无序、不可预测。

### 决策

使用 `uuid4().hex[:8]` 生成8位随机十六进制字符串作为任务ID。

### 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **UUID4截取** | 全局唯一、不可预测、短小 | 理论上有碰撞可能 |
| **完整UUID4** | 绝对唯一 | 太长、不便输入 |
| **递增整数** | 简单、有序、短 | 可预测、分布式困难 |
| **时间戳+随机** | 有序、唯一性好 | 实现复杂 |

### 后果

**正面影响：**
- ID短小，便于命令行输入
- 全局唯一性
- 不可预测，一定程度保证安全

**负面影响：**
- 理论上存在碰撞可能（概率极小）
- 无序，不便排序

**风险评估：**
- 8位十六进制 = 2^32 种可能 ≈ 43亿
- 按生日悖论，约65536个任务时碰撞概率达50%
- 对于个人使用场景，风险可接受

**负面影响：**
- 代码相对繁琐
- 扩展复杂命令时配置较多

## ADR-007: 实现子任务功能

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

用户需要将大任务分解为小的可跟踪子项。

### 决策

在 `Task` 模型中添加 `subtasks` 字段，使用独立的 `SubTask` 数据类。

```python
@dataclass
class SubTask:
    title: str
    done: bool = False
    subtask_id: str = field(default_factory=lambda: uuid4().hex[:6])

@dataclass  
class Task:
    # ... 其他字段
    subtasks: list[SubTask] = field(default_factory=list)
```

### 替代方案

1. **嵌套任务** - 任务包含子任务，子任务可再包含子任务
2. **简单列表** - 只用字符串列表表示子任务
3. **独立任务实体** - 子任务也是完整的任务，通过父子关系关联

### 后果

**正面影响：**
- 实现简单，易于理解
- 支持进度计算
- 序列化友好

**负面影响：**
- 不支持多层嵌套
- 子任务功能相对简单

**未来扩展：**
- 可以考虑支持多层嵌套
- 添加子任务优先级、截止日期等属性

## ADR-009: 引入事件驱动架构

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

随着功能增加，需要在任务状态变更时触发多种响应（如通知、日志、统计），传统的紧耦合方式难以扩展。

### 决策

引入事件驱动架构：
- 创建 `EventBus` 作为中央事件调度器
- 定义 `EventType` 枚举表示系统事件
- 任务操作触发相应事件
- 插件通过监听事件实现功能扩展

```python
class EventBus:
    def subscribe(self, event_type: EventType, handler: Callable)
    def emit(self, event: Event)
```

### 替代方案

1. **回调函数** - 在关键方法中接受回调参数
2. **装饰器模式** - 用装饰器包装核心方法
3. **钩子系统** - 在预定义位置调用注册的钩子函数
4. **消息队列** - 使用外部消息队列（Redis/RabbitMQ）

### 后果

**正面影响：**
- 核心业务逻辑与扩展功能解耦
- 支持插件系统的基础设施
- 易于添加新的事件响应逻辑
- 事件历史可用于调试和审计

**负面影响：**
- 增加系统复杂度
- 同步事件处理可能影响性能
- 事件处理异常需要特殊处理

**缓解措施：**
- 保持事件处理器轻量级
- 对事件处理异常进行隔离
- 考虑未来引入异步事件处理

## ADR-010: 实现插件架构

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

需要支持功能扩展而不修改核心代码，如通知、导出、集成第三方服务等。

### 决策

基于事件系统实现插件架构：
- 定义 `BasePlugin` 抽象基类
- 创建 `PluginRegistry` 管理插件生命周期
- 插件通过监听事件实现功能
- 提供配置注入机制

```python
class BasePlugin(ABC):
    @abstractmethod
    def activate(self) -> None
    
    def deactivate(self) -> None
        pass
```

### 替代方案

1. **入口点机制** - 使用setuptools entry_points
2. **模块导入** - 动态导入Python模块
3. **配置文件驱动** - 通过配置文件定义扩展
4. **微服务架构** - 通过HTTP API集成外部服务

### 后果

**正面影响：**
- 核心功能与扩展功能分离
- 支持运行时插件管理
- 插件开发有标准接口
- 便于测试和调试

**负面影响：**
- 插件间可能存在依赖关系
- 插件异常可能影响系统稳定性
- 没有插件隔离机制

**缓解措施：**
- 插件异常不影响主流程
- 提供插件开发最佳实践文档
- 考虑未来引入插件沙箱

## ADR-011: 添加任务模板系统

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

用户经常创建类似的任务（如Bug修复、功能开发、版本发布），手动输入重复信息效率低下。

### 决策

实现任务模板系统：
- 创建 `TaskTemplate` 数据类定义模板结构
- 支持变量插值（`{variable}` 语法）
- 提供内置模板（bug-fix、feature、release）
- 支持自定义模板的持久化存储

```python
@dataclass
class TaskTemplate:
    name: str
    title_pattern: str
    description_pattern: str
    subtask_titles: list[str]
    variables: dict[str, str]
```

### 替代方案

1. **简单别名系统** - 为常用命令创建别名
2. **配置文件预设** - 在配置文件中定义预设参数
3. **脚本生成** - 用Shell脚本生成复杂任务
4. **模板引擎** - 使用Jinja2等模板引擎

### 后果

**正面影响：**
- 提高创建常见任务的效率
- 标准化团队工作流程
- 减少人为错误
- 支持复杂任务结构的快速创建

**负面影响：**
- 增加系统复杂度
- 模板维护需要额外工作
- 变量插值功能有限

**缓解措施：**
- 提供丰富的内置模板
- 保持模板语法简单直观
- 支持模板的导入导出

## ADR-012: 实现批量操作功能

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

用户需要对多个任务执行相同操作，如批量完成、批量分配、数据导入导出等。

### 决策

创建 `BatchOperations` 类提供批量操作：
- 批量状态变更（完成、取消、重新分配）
- 任务导入导出（JSON格式）
- 重复任务检测和清理
- 所有操作触发相应事件

```python
class BatchOperations:
    def complete_all(self, task_ids: list[str]) -> int
    def import_from_json(self, file_path: str) -> list[Task]
    def find_duplicates(self, threshold: float) -> list[tuple[Task, Task]]
```

### 替代方案

1. **Shell脚本** - 通过脚本调用多次CLI命令
2. **配置文件** - 通过配置文件定义批量操作
3. **SQL风格语法** - 实现类似SQL的批量操作语法
4. **图形界面** - 提供GUI进行批量选择操作

### 后果

**正面影响：**
- 提高批量操作效率
- 支持数据迁移和备份
- 帮助发现和清理重复数据
- 操作与事件系统集成

**负面影响：**
- 批量操作可能影响性能
- 错误操作的影响范围更大
- 重复检测算法准确性有限

**缓解措施：**
- 提供操作确认机制
- 支持操作预览功能
- 实现操作日志和回滚

## ADR-013: 标签系统实现

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

用户需要对任务进行分类和组织，现有的状态和优先级字段不足以满足复杂的分类需求。

在 `Task` 模型中添加 `tags` 字段：

@dataclass

### 后果

**正面影响：**
- 提供灵活的任务分类方式
- 支持多维度的任务组织
- 实现简单，序列化友好

**负面影响：**
- 标签管理功能有限
- 没有标签层次结构
- 标签重命名和合并较困难

**未来扩展：**
- 实现标签的统计和分析
- 支持标签的自动补全
- 添加标签管理命令

## 技术债务和改进计划

### v0.3.0 引入的新限制

1. **事件系统限制**
   - 同步处理可能影响性能
   - 事件历史仅在内存中存储
   - 缺少事件序列化和持久化

2. **插件系统限制**
   - 没有插件沙箱隔离
   - 插件配置管理简单
   - 缺少插件依赖管理

3. **模板系统限制**
   - 变量插值功能基础
   - 不支持条件逻辑和循环
   - 模板验证机制不足

4. **批量操作限制**
   - 重复检测算法简单
   - 批量操作缺少进度反馈
   - 没有操作回滚机制

### 当前系统限制（延续）

1. **性能问题** - JSON文件全量加载，不适合大数据集
2. **并发安全** - 缺少文件锁，多进程访问可能冲突
3. **查询能力** - 只支持简单筛选，缺少复杂查询
4. **配置管理** - 缺少配置文件和环境变量支持

### 改进方向

1. **短期优化：**
   - 实现异步事件处理
   - 添加事件持久化机制
   - 完善插件配置管理
   - 改进模板变量插值功能

2. **中期目标：**
   - 支持SQLite存储后端
   - 实现插件沙箱机制
   - 添加操作历史和回滚功能
   - 实现增量加载和缓存

3. **长期规划：**
   - Web界面支持
   - 多用户和权限管理
   - 实时协作功能
   - 企业级插件市场
