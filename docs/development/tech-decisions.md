# 技术决策记录

本文档记录 TaskManager 项目的重要技术决策和架构选择。

## ADR-001: 采用分层架构模式

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

需要设计一个可扩展、易维护的任务管理系统架构。

### 决策

采用经典的分层架构模式：
- 表示层 (CLI)
- 业务层 (Models)
- 存储层 (Storage)
- 工具层 (Stats)

### 后果

**正面影响：**
- 职责清晰，模块解耦
- 便于单元测试
- 支持多种存储后端
- 易于扩展新功能

**负面影响：**
- 代码层次较多
- 简单功能可能过度设计

**缓解措施：**
- 保持接口简洁
- 避免过度抽象

## ADR-002: 使用抽象基类定义存储接口

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

需要支持多种存储后端（内存、JSON文件、可能的数据库），同时保持业务逻辑与存储实现解耦。

### 决策

使用 Python ABC (Abstract Base Class) 定义 `BaseTaskStore` 接口：

```python
class BaseTaskStore(ABC):
    @abstractmethod
    def add(self, task: Task) -> str: ...
    @abstractmethod
    def get(self, task_id: str) -> Optional[Task]: ...
    # 其他抽象方法...
```

### 替代方案

1. **Protocol类型** - 更轻量，但缺少运行时检查
2. **鸭子类型** - 灵活但缺少接口约束
3. **直接继承** - 简单但不够灵活

### 后果

**正面影响：**
- 强制接口一致性
- 运行时类型检查
- IDE支持更好
- 便于Mock测试

**负面影响：**
- 增加代码复杂度
- 必须实现所有抽象方法

## ADR-003: 选择JSON作为默认持久化格式

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

需要选择一种数据持久化格式，要求简单、可读、跨平台兼容。

### 决策

使用JSON格式存储任务数据到本地文件。

### 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **JSON** | 可读性好、跨平台、标准库支持 | 性能一般、不支持复杂查询 |
| **SQLite** | 功能强大、查询灵活、ACID | 学习成本高、过于复杂 |
| **Pickle** | 性能好、Python原生 | 不可读、安全风险、不跨语言 |
| **YAML** | 可读性极好、支持注释 | 解析复杂、性能差 |
| **CSV** | 简单、Excel兼容 | 结构限制、不支持嵌套 |

### 后果

**正面影响：**
- 数据可读性好，便于调试
- 跨平台兼容
- 标准库支持，无额外依赖
- 容易备份和版本控制

**负面影响：**
- 大数据集性能不佳
- 不支持复杂查询
- 并发写入可能有问题

**缓解措施：**
- 数据量大时可切换到SQLite
- 实现文件锁防止并发问题

## ADR-004: 采用dataclass定义数据模型

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

需要定义任务相关的数据结构，要求类型安全、易于序列化。

### 决策

使用 Python 3.7+ 的 `@dataclass` 装饰器定义数据模型：

```python
@dataclass
class Task:
    title: str
    status: TaskStatus = TaskStatus.TODO
    priority: Priority = Priority.MEDIUM
    created_at: datetime = field(default_factory=datetime.now)
```

### 替代方案

1. **普通类** - 需要手写大量样板代码
2. **namedtuple** - 不可变，缺少方法支持
3. **dict** - 缺少类型约束和IDE支持
4. **Pydantic** - 功能强大但增加依赖

### 后果

**正面影响：**
- 自动生成 `__init__`、`__repr__` 等方法
- 类型注解支持
- IDE 自动完成
- 代码简洁

**负面影响：**
- Python 3.7+ 要求
- 一些高级特性需要手动实现

## ADR-005: 使用UUID作为任务ID生成策略

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

任务需要唯一标识符，要求全局唯一、无序、不可预测。

### 决策

使用 `uuid4().hex[:8]` 生成8位随机十六进制字符串作为任务ID。

### 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **UUID4截取** | 全局唯一、不可预测、短小 | 理论上有碰撞可能 |
| **完整UUID4** | 绝对唯一 | 太长、不便输入 |
| **递增整数** | 简单、有序、短 | 可预测、分布式困难 |
| **时间戳+随机** | 有序、唯一性好 | 实现复杂 |

### 后果

**正面影响：**
- ID短小，便于命令行输入
- 全局唯一性
- 不可预测，一定程度保证安全

**负面影响：**
- 理论上存在碰撞可能（概率极小）
- 无序，不便排序

**风险评估：**
- 8位十六进制 = 2^32 种可能 ≈ 43亿
- 按生日悖论，约65536个任务时碰撞概率达50%
- 对于个人使用场景，风险可接受

## ADR-006: 选择argparse构建CLI接口

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

需要构建用户友好的命令行接口。

### 决策

使用Python标准库的 `argparse` 模块构建CLI。

### 替代方案

| 方案 | 优点 | 缺点 |
|------|------|------|
| **argparse** | 标准库、功能完整、文档详细 | 语法繁琐、不够现代 |
| **click** | 装饰器语法、功能丰富 | 外部依赖、学习成本 |
| **typer** | 基于类型提示、现代 | 新项目、外部依赖 |
| **sys.argv** | 简单直接 | 功能有限、解析复杂 |

### 后果

**正面影响：**
- 无外部依赖
- 功能完整，支持子命令
- 自动生成帮助信息
- 错误处理完善

**负面影响：**
- 代码相对繁琐
- 扩展复杂命令时配置较多

## ADR-007: 实现子任务功能

**日期：** 2026-03-06  
**状态：** 已采纳  
**决策者：** 开发团队  

### 背景

用户需要将大任务分解为小的可跟踪子项。

### 决策

在 `Task` 模型中添加 `subtasks` 字段，使用独立的 `SubTask` 数据类。

```python
@dataclass
class SubTask:
    title: str
    done: bool = False
    subtask_id: str = field(default_factory=lambda: uuid4().hex[:6])

@dataclass  
class Task:
    # ... 其他字段
    subtasks: list[SubTask] = field(default_factory=list)
```

### 替代方案

1. **嵌套任务** - 任务包含子任务，子任务可再包含子任务
2. **简单列表** - 只用字符串列表表示子任务
3. **独立任务实体** - 子任务也是完整的任务，通过父子关系关联

### 后果

**正面影响：**
- 实现简单，易于理解
- 支持进度计算
- 序列化友好

**负面影响：**
- 不支持多层嵌套
- 子任务功能相对简单

**未来扩展：**
- 可以考虑支持多层嵌套
- 添加子任务优先级、截止日期等属性

## 技术债务和改进计划

### 当前限制

1. **性能问题** - JSON文件全量加载，不适合大数据集
2. **并发安全** - 缺少文件锁，多进程访问可能冲突
3. **查询能力** - 只支持简单筛选，缺少复杂查询
4. **配置管理** - 缺少配置文件和环境变量支持

### 改进方向

1. **短期优化：**
   - 添加文件锁机制
   - 实现配置文件支持
   - 优化搜索算法

2. **中期目标：**
   - 支持SQLite存储后端
   - 实现增量加载
   - 添加索引和缓存

3. **长期规划：**
   - Web界面支持
   - 多用户和权限管理
   - 任务协作功能
